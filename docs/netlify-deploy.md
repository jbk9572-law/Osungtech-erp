# Netlify에 백업 배포로 올려서 테스트하기

지금 Vultr 서버(Docker + Caddy, 월 $5)로 운영 중인 걸 끄지 않고, Netlify
무료 플랜에 같은 코드를 병렬로 배포해서 안정적으로 쓸 수 있는 수준인지
먼저 확인하는 절차다. 확인되면 도메인만 옮겨서 Vultr를 정리하고 비용도
아낀다. 도메인은 테스트가 끝나고 확신이 선 뒤에만 옮긴다.

## 1. Netlify에 사이트 추가

1. https://app.netlify.com 가입(무료, GitHub 계정으로 바로 가능) →
   **Add new site > Import an existing project**
2. GitHub 연동 후 `jbk9572-law/Osungtech-erp` 저장소 선택
3. 배포할 브랜치는 `main`으로 지정 (지금 Vultr에서 운영 중인 것과 같은 코드)
4. Build settings는 저장소에 있는 `netlify.toml`을 그대로 인식한다 —
   빌드 명령(`npm run build`)이 자동으로 채워지면 그대로 두면 된다

## 2. 환경변수 등록 (필수)

Site configuration > **Environment variables**에서 아래 두 개를 등록한다
(Vultr 서버의 `.env.production`과 같은 값):

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

값은 Supabase 대시보드 > Project Settings > API에서 그대로 복사하면 된다.
이 값들은 빌드 시점에 번들에 박히는 공개 값이라 등록 안 하면 빌드는
되지만 로그인/DB 조회가 전부 실패한다.

### (선택) 넷리파이 사용량 위젯 보이게 하기

사이드바에 넷리파이 무료 플랜 대역폭 사용량을 보여주려면 아래도 등록한다:

- `NETLIFY_API_TOKEN`: Netlify 계정의 **User settings > Applications >
  Personal access tokens > New access token**에서 발급한 토큰
- `NETLIFY_TEAM_SLUG`: (선택) 팀이 여러 개면 지정, 하나뿐이면 안 넣어도 자동으로 찾는다

등록 안 해도 앱 사용에는 아무 문제 없다 — 이 위젯만 안 보일 뿐이다.

## 3. 첫 배포 확인

배포가 끝나면 Netlify가 자동으로 만들어주는 임시 주소
(`https://<사이트이름>.netlify.app`)로 접속해서 확인한다:

- 로그인이 되는지
- 대시보드/매입/매출/할일 목록이 실제 데이터로 뜨는지
- 매입·매출 등록, 재고 반영이 정상 동작하는지 (될 수 있으면 테스트성
  거래처/품목으로 시험 등록해보고 지워도 되는 것으로 확인)
- 모조지 계산, 인쇄(명세표/출고증) 화면도 열어본다
- 엑셀 다운로드(내보내기)/업로드(일괄등록) — 데이터가 많을 때 넷리파이
  무료 플랜의 함수 실행시간 제한(동기 함수 기준 약 10초)에 걸리는지 확인

이 단계에서는 아직 도메인을 연결하지 않으므로, `.netlify.app` 임시
주소로만 확인하면 된다 — Vultr 서버 쪽 실제 서비스는 전혀 영향받지
않는다. **주의**: Netlify 앱과 Vultr 앱은 같은 Supabase DB를 보고 있어서
(따로 분리된 DB가 아니다), 넷리파이에서 등록한 테스트 데이터도 Vultr
쪽 실서비스 화면에 그대로 보인다 — 테스트 후에는 지워두는 게 좋다.

## 4. 도메인은 나중에

지금 쓰는 도메인은 Vultr 서버(Caddy)를 그대로 가리키고 있다. Netlify
쪽이 며칠 이상 안정적으로 확인되고 나서, 이 도메인의 DNS를 Netlify로
옮기면 그때부터 실제 서비스가 Netlify로 넘어간다. 그 전까지는 Vultr
서버를 끄지 않는다. 전환이 끝나면 Vultr 서버는 해지해서 월 $5를 아낄 수
있다.

## 참고: 코드에서 미리 손봐둔 부분

- `next.config.ts`의 `output: "standalone"`은 Vultr Docker 배포 전용
  설정이라 Netlify 빌드에서는 자동으로 꺼지도록 해뒀다(Netlify가 빌드 중
  자동으로 심어주는 `NETLIFY` 환경변수로 판별 — 이 변수는 빌드 시점에만
  설정되지만 `next.config.ts`는 빌드 시점에 읽히므로 문제없다). Vultr
  쪽 빌드에는 영향 없다.
- 사이드바의 "서버 디스크" 위젯은 실제 서버 파일시스템을 읽는 코드라
  Vultr 같은 상시 서버에서만 의미가 있다. Netlify(서버리스)에서는 이
  값 자체가 존재하지 않아서(요청마다 뜨는 임시 컨테이너라 "그 서버의
  디스크"라는 개념이 없다) 위젯을 아예 숨기고, 대신 넷리파이 계정의
  대역폭 사용량 위젯을 보여준다(위 "선택" 항목 참고). 이 판별은 요청을
  처리하는 함수 런타임에도 항상 존재하는 넷리파이 예약 변수 `SITE_ID`로
  한다(`NETLIFY` 변수는 빌드 시점 전용이라 런타임에는 비어있어서 판별에
  못 쓴다). 빌드 시간(분) 사용량은 넷리파이가 공개 API로 제공하지 않아서
  대역폭만 다룬다 — 정확한 값이 없는 건 안 보여주는 쪽을 택했다.
  참고로 2025년 9월 이후 신규 가입 계정은 대역폭 전용 고정 한도가 없고
  빌드/함수/대역폭이 매달 300 크레딧을 나눠쓰는 구조라, 이런 계정에서는
  퍼센트 막대 없이 사용량(MB)만 보여준다(고정 한도가 있는 레거시 계정은
  기존처럼 퍼센트 막대로 보인다).
- 파일 업로드(브랜딩 이미지, 메신저 첨부파일)는 이미 Supabase Storage를
  쓰고 있어서 플랫폼이 바뀌어도 그대로 동작한다.
